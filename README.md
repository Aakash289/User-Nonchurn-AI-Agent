# User NonChurn AI

User NonChurn AI is an autonomous analytics agent that ingests raw product event logs, identifies the statistically derived Activation Signal - the specific user action most correlated with long-term retention - and segments new users by their likelihood of churning. Everything is surfaced through an interactive Streamlit dashboard.

The core question this agent answers: "What is the one thing a new user must do in their first 7 days to predict they will stick around?" And it answers it with data, not gut feel.

---

## What It Does

Most product teams guess their activation metric. They pick "created a project" or "added a task" because it feels like engagement. User NonChurn AI tests every observable behaviour statistically and tells you which one actually predicts retention, and by how much.

The agent reads raw event logs with no pre-assigned labels, engineers 7 behavioural features per user from their first 7 days, derives a retention label purely from whether the user came back after Day 13, runs Pearson correlations between every feature and retention, defines the Activation Signal as the collaboration event with the highest business relevance, scores every user from 0 to 1, segments them into High Activation, At Risk, or Likely Churned, and recommends a specific intervention for each segment.

## The Data

### events.csv

This file is the raw event log and the only input the agent ever sees. It mimics exactly what a real SaaS product database would contain: a timestamped record of every action every user took.

The file spans 60 days with 120 users (2 signups per day) and approximately 1,400 events total. Events after Day 13 from a user's signup date are used exclusively to determine whether that user was retained. The agent never uses late events to build features.

### Fields

| Field | Type | Description |
|-------|------|-------------|
| user_id | string | Unique identifier for each user, e.g. usr_001 |
| user_name | string | Display name of the user |
| event | string | Name of the action taken |
| timestamp | datetime | Exact date and time the action occurred |

### Event Types

| Event | What It Means |
|-------|---------------|
| signed_up | User created an account - Day 0 anchor |
| created_workspace | User set up their workspace |
| created_project | User started a new project |
| added_task | User added a task to a project |
| invited_teammate | User sent an invite to a colleague |
| teammate_joined | The invited colleague accepted and joined |
| teammate_commented | Teammate left a comment on a task |
| assigned_task | A task was assigned to a team member |
| teammate_added_task | A teammate added their own task |
| logged_in | User returned and opened the app |
| viewed_onboarding | User watched the intro tutorial |

### Behaviour Patterns

The data generator creates users across 9 realistic behaviour patterns.

| Pattern | Description | Retained |
|---------|-------------|----------|
| A | Invites team, collab within 48h, strong engagement | Yes |
| B | Invites team, teammate joins late outside 48h window | No |
| C | Solo user, creates content, never invites anyone | No |
| D | Lurker, watches onboarding, logs in twice | No |
| E | Ghost, signs up and disappears | No |
| F | Power user, multiple invites, heavy collaboration | Yes |
| G | Noise - hits activation signal but still churns | No |
| H | Noise - solo user who comes back without activation | Yes |
| I | Noise - lurker who returns after re-engagement | Yes |

Patterns G, H, and I exist specifically to introduce realistic noise. Without noise, the agent produces perfect correlations (r=1.0, lift=∞) which look impressive but do not reflect real product data. With noise, you get realistic numbers: r≈0.485 for the activation signal (collab_within_48h), lift≈3.74×, retention if triggered around 68%, and retention if missed around 18%. The top raw correlating feature is typically tasks_added (r≈0.557), but Claude chooses collab_within_48h as the activation signal because it is more actionable.

## What The Agent Does

The agent reads only events.csv. It has no knowledge of which pattern a user belongs to and no pre-assigned retention labels. It figures everything out from the raw event log.

### Feature Engineering

For each user, the agent looks only at events within Day 0 to Day 7 from their signup timestamp. It computes 7 features.

| Feature | How It Is Computed |
|---------|-------------------|
| projects_created | Count of created_project events in first 7 days |
| tasks_added | Count of added_task events in first 7 days |
| teammates_invited | Count of invited_teammate events in first 7 days |
| teammate_joined | 1 if any teammate_joined event exists in first 7 days |
| collab_within_48h | 1 if any collab event occurred within 48 hours of the first invite |
| sessions | Count of distinct calendar days with any activity in first 7 days |
| watched_onboarding | 1 if viewed_onboarding appears in first 7 days |

### Retention Label

The agent checks whether the user had any event after Day 13 from their signup. If yes, retained equals 1. If no, retained equals 0. No other information is used. This label is derived purely from behaviour in the raw log, not from any external source.

### Scoring Formula

Each user receives an activation score between 0 and 1.

```
score = collab_within_48h  × 0.50
      + teammate_joined     × 0.25
      + min(invites, 3) / 3 × 0.15
      + min(sessions, 7) / 7 × 0.10
```

Weights reflect correlation strength. collab_within_48h gets the highest weight because the collaboration event is the most specific and actionable predictor. Sessions gets the lowest weight because high session counts also appear in solo users who churn.

### Segmentation

| Score Range | Segment | What It Means |
|-------------|---------|---------------|
| 0.65 and above | High activation | User completed the social loop - upsell candidate |
| 0.30 to 0.65 | At risk | Invited someone but collab has not closed - one nudge away |
| Below 0.30 | Likely churned | Solo or inactive - needs win-back or exclusion |

### Activation Signal

The Activation Signal is fixed as collab_within_48h - the moment a user invites a teammate AND that teammate takes an action within 48 hours. This is the business-defined trigger the product team can directly engineer. The agent also surfaces the top raw correlating feature separately so you can track if another behaviour is emerging as a stronger signal over time.

### Output: agent_results.json

The agent saves all computed results to agent_results.json. This file contains summary counts of total users, events, and segment sizes, activation signal stats including retention rates, lift, and Pearson r, the full correlation table for all 7 features, and per-user records with score, segment, recommended action, and day-by-day score history.

## The Dashboard

### KPI Cards

Four cards at the top give an instant read on the cohort. Activated shows users scoring 0.65 or above as a count and percentage of the cohort. At risk shows users scoring 0.30 to 0.65. Likely churned shows users below 0.30. Top correlation shows the single feature with the highest absolute Pearson r against the retained label across all users.

### Activation Signal Banner

The most important section of the dashboard. It shows the statistically derived definition of the activation moment with four metric boxes.

Retention if triggered is the percentage of users who hit the activation signal and came back after Day 14. A higher number means the signal is a stronger predictor.

Retention if missed is the percentage of users who never hit the signal but still returned. A lower number means the signal is more exclusive and separates retained from churned users more cleanly.

Lift is calculated as Retention if triggered divided by Retention if missed. It tells you how many times more likely a signal-hit user is to retain compared to a miss. A lift of 3.74× means users who hit the activation signal are more than 3 times more likely to come back than those who did not.

Correlation (r) is the Pearson correlation coefficient between the activation signal feature and the retained label across all 120 users. The range is −1 to +1. Above 0.5 is considered strong in behavioural analytics. The activation signal (collab_within_48h) has a Pearson r of 0.485. The top raw correlating feature is tasks_added at r=0.557, but Claude chooses collab_within_48h as the activation signal because it directly measures the collaboration moment, which is more actionable than task volume.

### Feature Correlations Chart

A horizontal bar chart ranking all 7 features by their absolute Pearson r with the retained label. Bar width represents the strength of the relationship. Colour represents direction and magnitude. Green bars indicate strong positive predictors such as collab events and teammate joined. Blue bars indicate moderate positive predictors such as sessions and invites. Gray bars indicate weak predictors such as tasks added and projects created - almost everyone does these regardless of whether they retain. Red bars indicate negative predictors. Watched onboarding is negative because users who watch the tutorial are typically confused, not curious, making it a churn signal rather than an engagement signal.

### Segment Breakdown

Proportional bars showing how the cohort splits across the three segments, with recommended actions listed for each. High activation users should see an upgrade prompt immediately. At risk users need their invitee nudged with a single email. Likely churned users should receive a win-back email at 72 hours and be excluded from engagement budgets after 7 days if unresponsive.

### User Activation Score Build-Up Chart - Day 0 to Day 7

This chart is the core predictive view of the dashboard. It shows how a selected user's activation score accumulated day by day across their first 7 days only.

The reason the chart shows only 7 days and not the full 60 is fundamental to the purpose of the agent. The agent is trying to predict future behaviour from early signals. Day 0 to Day 7 is the prediction window. Everything after Day 13 is what we are trying to predict. If the chart showed all 60 days it would include the very return events it is trying to forecast, which would make it a description of what happened rather than a prediction of what will happen.

A flat line near zero means the user did no qualifying actions that day - no invites, no collaboration, and a low session count. A sudden jump upward means a key event fired. The most significant jump is when collab_within_48h triggers, which adds 0.50 to the score in a single step. A smaller jump of 0.25 occurs when a teammate joins. The green dashed line at 0.65 is the activation threshold. Any user whose score line crosses it is predicted to retain. The amber dotted line at 0.30 marks the at-risk boundary. Users between the two lines need an intervention. Users whose lines stay flat near zero all week are predicted churners.

Hovering over any point on the chart shows exactly which events fired that day. Expanding the table below the chart gives a full day-by-day breakdown of events and cumulative scores.

### All Users Table

A sortable and filterable table showing every user with their activation score, segment, recommended action, collaboration flag, invite count, session count, and retention outcome. The segment filter at the top lets you isolate just the at-risk users, which is the highest-leverage action list for any given day.

### Raw Event Log Explorer

A dropdown selector lets you pick any user and see their complete event timeline directly from events.csv displayed as a colour-coded sequence. Blue dots represent setup events such as signing up and creating a workspace. Purple dots represent solo work such as creating projects and adding tasks. Orange dots represent social attempts such as inviting a teammate and the teammate joining. Green dots represent confirmed collaboration such as a teammate commenting or a task being assigned - this is the activation moment. Gray dots represent return visits. Red dots represent the onboarding video, which is a confusion signal.

A user who activates will show a clear progression from blue to purple to orange to green. A churned user's timeline stops at blue or purple and never reaches orange or green.

## Interpreting The Results

A high lift above 3× means the activation signal is very selective and hitting it strongly predicts retention. A low lift between 1 and 2× means multiple paths to retention exist and the signal matters but is not the only one. A negative onboarding correlation means users who watch tutorials are more confused than curious and onboarding should be rethought. A high retention-if-missed percentage means solo users find value without collaboration and a separate solo use case may be worth supporting. A large at-risk segment means many users are one teammate action away from activating and email nudges to invitees are the highest-priority growth action.

## Analytics and Calculations

This section explains every formula and statistical method the agent uses, step by step.

---

### Feature Engineering

For each user, the agent looks only at events within Day 0 to Day 7 from their signup timestamp. No events outside this window are used to build features.

The signup timestamp is the anchor. It is found by locating the `signed_up` event in each user's event history. From this anchor, every event gets a day offset:

```
day_offset = (event_timestamp - signup_timestamp).days
```

Events with day_offset <= 7 are included in the feature window. Events at day_offset >= 14 are used exclusively to derive the retention label.

#### The 7 Features

| Feature | Formula | Predictive Power |
|---|---|---|
| projects_created | COUNT(created_project events in Day 0-7) | Weak - both retained and churned users create a project on Day 0 |
| tasks_added | COUNT(added_task events in Day 0-7) | Moderate - solo churned users also add tasks in week one |
| teammates_invited | COUNT(invited_teammate events in Day 0-7) | Strong - shows social intent |
| teammate_joined | 1 if any teammate_joined event in Day 0-7, else 0 | Strong - intent became action |
| collab_within_48h | 1 if collab event within 48h of first invite, else 0 | Strongest - the Activation Signal |
| sessions | COUNT(DISTINCT calendar dates with any event in Day 0-7) | Moderate - engagement frequency |
| watched_onboarding | 1 if viewed_onboarding in Day 0-7, else 0 | Negative - confusion signal |

#### The Activation Signal Formula

collab_within_48h is the most complex feature. It checks whether any collaboration event occurred within exactly 48 hours of the first invite being sent. Collaboration events are: `teammate_commented`, `assigned_task`, `teammate_added_task`.

```
invite_ts         = MIN(timestamp WHERE event = 'invited_teammate' AND day_offset <= 7)
collab_delta      = (collab_event_timestamp - invite_ts).total_seconds()
collab_within_48h = 1 IF collab_delta <= 172800 ELSE 0
```

172,800 = 48 hours x 3,600 seconds per hour.

**Example - hit:** invite at 14:22, teammate_commented at 15:10 same day. Delta = 2,880 seconds. 2,880 <= 172,800 so collab_within_48h = 1.

**Example - miss:** invite at 10:00 on Day 3, teammate_joined at 14:00 on Day 5, no collab event ever. collab_within_48h = 0.

### Retention Label Derivation

The retention label is not pre-assigned. The agent derives it from the raw event log:

```
retained = 1  IF  COUNT(events WHERE day_offset >= 14) > 0
retained = 0  OTHERWISE
```

Day 14 is the threshold because it is far enough from signup to confirm genuine return behaviour, but early enough that at-risk users can still be intercepted. If we waited until Day 30, the intervention window would have closed.

---

### Pearson Correlation

Pearson r measures the strength and direction of the linear relationship between each feature and the retention label across all users. It always falls between -1 and +1.

#### The Formula

```
r = SUM( (xi - mean_x) * (yi - mean_y) )
    ─────────────────────────────────────────────────────
    SQRT( SUM((xi - mean_x)^2) * SUM((yi - mean_y)^2) )
```

Where xi is the feature value for user i and yi is the retention label (0 or 1) for user i.

In Python:

```python
def correlation(xs, ys):
    n  = len(xs)
    mx = sum(xs) / n
    my = sum(ys) / n
    num = sum((x-mx)*(y-my) for x, y in zip(xs, ys))
    dx  = sqrt(sum((x-mx)**2 for x in xs) or 1e-9)
    dy  = sqrt(sum((y-my)**2 for y in ys) or 1e-9)
    return round(num / (dx * dy), 3)
```

The `1e-9` floor prevents division by zero if all values in a column are identical.

#### What Different r Values Mean

| r value | Interpretation |
|---|---|
| +0.7 to +1.0 | Strong positive - users who did this are much more likely to retain |
| +0.4 to +0.7 | Moderate positive - meaningful relationship with exceptions |
| +0.1 to +0.4 | Weak positive - small tendency, many exceptions |
| 0.0 | No relationship - this action tells us nothing about retention |
| -0.1 to -0.4 | Weak negative - slight tendency toward churn |
| -0.4 to -1.0 | Strong negative - users who did this are more likely to churn |

#### Worked Example

With 6 users and collab_within_48h as the feature:

```
User     collab(x)   retained(y)   x-mean_x   y-mean_y   product
Sarah         1           1          +0.5        +0.5       +0.25
Marcus        0           0          -0.5        -0.5       +0.25
James         1           1          +0.5        +0.5       +0.25
Priya         0           0          -0.5        -0.5       +0.25
Tom           1           1          +0.5        +0.5       +0.25
Lena          0           0          -0.5        -0.5       +0.25

mean_x = 0.5   mean_y = 0.5   SUM(products) = 1.50
dx = SQRT(0.5^2 x 6) = 1.225
dy = SQRT(0.5^2 x 6) = 1.225
r  = 1.50 / (1.225 x 1.225) = 1.0 (perfect, no noise)
```

With 120 users including noise patterns G, H, and I, the correlation drops to around 0.45 to 0.55 because some retained users never hit the signal (pattern H) and some churned users did hit it (pattern G).

### Lift

Lift answers: how many times more likely is a user who hit the activation signal to retain, compared to a user who did not?

```
retention_rate_hit  = retained_users_who_hit_signal   / total_users_who_hit_signal
retention_rate_miss = retained_users_who_missed_signal / total_users_who_missed_signal
lift = retention_rate_hit / retention_rate_miss
```

**Example from a typical run:**

```
retention_rate_hit  = 68%
retention_rate_miss = 18%
lift = 68 / 18 = 3.78x
```

A lift of 3.78x means a user who hit the activation signal is 3.78 times more likely to return than one who did not.

#### How to Interpret Lift

| Lift range | Interpretation |
|---|---|
| Below 1.5x | Weak signal - look for a better activation metric |
| 1.5x to 2.5x | Moderate - use alongside other signals |
| 2.5x to 4x | Strong - build onboarding flows around this moment |
| Above 4x | Very strong - make this your primary activation metric |

### Scoring Formula

The activation score is a weighted sum of normalised features, always between 0 and 1.

```
score = collab_within_48h        x 0.50
      + teammate_joined           x 0.25
      + min(invites, 3) / 3       x 0.15
      + min(sessions, 7) / 7      x 0.10
score = min(score, 1.0)
```

Weights are auto-normalised to sum to 1.0 before being applied. The score is hard-capped at 1.0.

#### Why Each Weight

| Feature | Weight | Reason |
|---|---|---|
| collab_within_48h | 0.50 | Highest weight. Most specific and actionable predictor. Binary so full weight when triggered. |
| teammate_joined | 0.25 | Strong predictor. Confirms invite was accepted. |
| teammates_invited | 0.15 | Moderate predictor. Shows social intent. Capped at 3 invites. |
| sessions | 0.10 | Weakest of the four. Present in both retained and churned users. |

#### Feature Normalisation

Binary features (collab_within_48h and teammate_joined) are already 0 or 1. Count features are normalised:

```
teammates_invited_normalised = min(teammates_invited, 3) / 3
sessions_normalised          = min(sessions, 7) / 7
```

#### Three Worked Examples

**High Activation (score 0.785):**
```
collab = 1   +0.500
joined = 1   +0.250
invites = 2  min(2,3)/3 x 0.15 = +0.100
sessions = 7 min(7,7)/7 x 0.10 = +0.100
score = 0.785  (segment: High Activation)
```

**At Risk (score 0.357):**
```
collab = 0   +0.000  (teammate never acted within 48h)
joined = 1   +0.250
invites = 1  min(1,3)/3 x 0.15 = +0.050
sessions = 4 min(4,7)/7 x 0.10 = +0.057
score = 0.357  (segment: At Risk)
```

**Likely Churned (score 0.057):**
```
collab = 0   +0.000
joined = 0   +0.000
invites = 0  +0.000
sessions = 4 min(4,7)/7 x 0.10 = +0.057
score = 0.057  (segment: Likely Churned)
```

### Segmentation Thresholds

```
IF score >= 0.65  ->  High Activation  (completed the social loop)
IF score >= 0.30  ->  At Risk          (social intent, loop not closed)
IF score <  0.30  ->  Likely Churned   (solo or inactive)
```

The 0.65 threshold represents the minimum score a user achieves if both collab_within_48h and teammate_joined fire, even with zero invites and minimal sessions. The 0.30 threshold captures users who invited someone and their teammate joined, but the collaboration loop never closed.

---

### Recommended Actions Logic

```python
if   score >= 0.65:              action = "Upsell - show upgrade prompt"
elif teammates_invited > 0:      action = "Nudge teammate - send reminder"
elif projects_created > 0:       action = "Show invite prompt in-app"
else:                            action = "Win-back email after 72h"
```

An at-risk user who invited a teammate gets a different action than one who never invited anyone, even if their scores are similar. The former needs one email to the invitee to close the loop. The latter needs a prompt to try inviting someone first.

---

### Day-by-Day Score Chart

The score build-up chart in the dashboard shows how the activation score grew day by day. It is computed by running the scoring formula incrementally:

```
score_day_N = score(events WHERE day_offset <= N)    for N in 0..7
```

The score on Day 0 uses only Day 0 events. Each day the score can only stay the same or increase. A sudden jump on Day 1 or Day 2 typically means collab_within_48h triggered, adding 0.50 in a single step. A flat line near zero all week means the user is predicted to churn.
